Введение:
Постарался разработать такую структуру тестового проекта, чтобы она была легко масштабируема.
Из-за этого на первый взгляд может показаться, что некоторые классы и методы избыточны, но при большом тестовом проекте они будут необходимы.
Требования и "HoTo" для запуска находятся в README.md файле в корне репозитария.

Структура автоматической системы:
lib\*.jar
  Библиотеки, которые используются в проекте.
bin\
  Папка со скомпилированными скриптами. Там же лежит "запускатор" для тестов. И web-драйвер для chrome-браузера для Windows-платформы.
src
  Папка с исходниками скриптов.
    com\core
      Тут лежат классы, которые будут являться общими для всех тестов текущих и будущих.
      CommonMethods.java - класс с различными общими методами работы с UI и вспомогательные.
      Logger.java - класс с методами логирования.
      MyWebDriver.java - класс для выбора web-драйвера для браузера. Тут предполагается добавлять поддержку дополнительных браузеров.
    com\data
      Тут предполагается хранить данные для тестов. Пока есть только данные для теста формы добавления приложения: DataAddApp.java
      Т.о. для текущего теста достаточно добавить нужные тест-кейсы в данный файл и они автоматически будут выполнены.
    com\testsuite\TestSuite.java
      Класс тест-набора, который агрегирует в себе все тест-кейсы. А также выполняет ряд подготовительных действий: создает драйвер - проходит аутентификацию - запускает тесты - уничтожает драйвер.
    com\testcases\*.java
      Тут лешат все тест-кейсы, которые вызываются из тест-набора. На текущий момент тут два больших тест-кейса:
      TestAddAppPositive.java - позитивные тесты, т.е. тесты, которые предполагают успешный результат, в нашем случае - создание приложения.
      TestAddAppNegative.java - негативные тесты, т.е. тесты, которые предполагают появление ошибки.

Что проверяется:
  1. Позитивные тесты: вводим различные данные в поля формы добавления нового App. Проверяем, что создаются Adplace с определенными вкладками и title. Проверяем, что показывается введенное ранее уникальное имя приложения. Проверяем, что показывается нужная иконка.
  2. Негативные тесты: вводим неправильные значения в форме. Ждем появления нужных сообщений об ошибках.
  
Что НЕ проверяется:
  1. Показ добавленного приложения в списке всех приложений. Этого не было в техзадании.
  2. В форму добавления нового app вводятся только значащие поля, т.е. те, от которых зависит отображение Adplace.
  Исследовательским путем удалось выяснить, что это поля: platformType, name, description, appIconFile, category.
  Добавление обработки новых полей, по моему мнению, не должно вызвать проблем. И состоит из двух этапов: добавление тестовых данных и добавление обработчика для этих данных.

Что хотелось бы доделать, но по "Контрольному Времени" не успел:
  1. Сделать нормальный обработчик ошибок. Сейчас, на мой взгляд, если есть ошибки в тестах они не очень удобно выводятся.
  2. Больший набор входных данных. Т.к. это демонстрационный проект, я не сильно задумывался о тестовом покрытии. Проверил наиболее критичные наборы входных даных.
  3. Сейчас тестовый скрипт работает только на Win-платформах. Чтобы запустить тесты на других платформах, нужно скачать драйвер для браузера на конкретной платформе и переписать cmd-файл в sh-файл, например.
  4. Возможно, тестовые данные нужно соотнести с тестом. Т.о. при выполнении 1-го теста брался только один набор тестовых данных. Это обсуждаемый момент, т.к. в этом случае будет избыток кода, но, возможно, будет удобнее такой код отлаживать. В общем, переделать под такую структуру много времени не займет.
  
Результаты работы. Выводы.
1. Очень не хватает id для элементов. Чтобы привязаться к контролу приходится использовать нетривиальные способы, это усложняет написание тестов. Верстальщик должен работать в паре с тестировщиком.
2. В коде теста inline встречаются комментарии с пометкой "TODO". В этом комментарии содержатся рекоммендации для улучшения теста.
3. Найденная проблема:
Если в форме добавления нового app НЕ заполнять name и category, но выбрать неправильную картинку (Больше, чем 200x200), то ошибка про картинку не показывается - показывается только ошибка о незаполненности name и category. Эту проблема вопроизводится на последнем тесте.
